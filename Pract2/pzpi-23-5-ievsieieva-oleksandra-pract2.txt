3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1.	 Загальні відомості про систему
	Система Spotify є яскравим прикладом сучасної високорозподіленої платформи, розробленої шведською компанією Spotify AB. Її призначення охоплює надання послуг потокового відтворення аудіоконтенту мільйонам користувачів по всьому світу. Основна функціональність системи базується не лише на відтворенні медіафайлів, а й на складних алгоритмах персоналізації, що вимагають обробки величезних масивів великих даних. Як глобальний сервіс, Spotify працює у багатьох регіонах, що накладає особливі вимоги до архітектури щодо затримок та доступності.

3.2.	Архітектурний стиль системи
	Архітектура Spotify базується на мікросервісному стилі, що став галузевим стандартом для великих систем. Вибір саме такої моделі обґрунтований необхідністю забезпечити повну автономність крос-функціональних команд (Squads). Кожна команда відповідає за окремий функціональний блок (наприклад, плейлисти або пошук), що дозволяє проводити сотні розгортань коду щодня без ризику зупинки всієї системи. Такий підхід забезпечує високу масштабованість, оскільки кожен мікросервіс може бути адаптований під специфічне навантаження незалежно від інших частин платформи.

3.3.	Основні компоненти системи
	Система розділена на декілька функціональних рівнів. Користувацький інтерфейс представлений мобільними та десктопними додатками, які використовують концепцію мікрофронтендів для відображення різних частин екрана. Серверна частина складається з тисяч мікросервісів, що взаємодіють через API Gateway. Рівень бізнес-логіки відповідає за складні процеси, такі як генерація щотижневих відкриттів за допомогою моделей машинного навчання. Рівень зберігання даних використовує гібридний підхід: для швидкого доступу до метаданих застосовується Google Cloud Bigtable, а для великих обсягів медіафайлів — розподілені хмарні сховища. Допоміжні сервіси забезпечують наскрізну авторизацію та моніторинг стану системи в реальному часі.

3.4.	Взаємодія компонентів
	Взаємодія між компонентами реалізована через поєднання синхронного gRPC та асинхронного обміну повідомленнями через Apache Kafka. Коли користувач додає трек у плейлист, сервіс плейлистів генерує подію, яку підхоплюють сервіси аналітики та рекомендацій. Це дозволяє системі бути подієво-орієнтованою (event-driven), забезпечуючи актуальність даних у всіх модулях без жорсткої прив'язки сервісів один до одного. Для координації тисяч внутрішніх API використовується платформа Backstage, яка дозволяє розробникам швидко знаходити та інтегрувати існуючі сервіси. 3.5.	Масштабованість та надійність
	Балансування навантаження: Використання сервісної сітки (Service Mesh) та Envoy для розумного розподілу трафіку.
	Відмовостійкість: Принцип «Graceful degradation» — якщо сервіс рекомендацій не відповідає, користувач бачить стандартний список треків замість помилки.
	Кешування: Використання CDN для доставки медіафайлів з мінімальною затримкою залежно від регіону.
	Контейнеризація: Робота всіх сервісів у середовищі Kubernetes (GKE) для швидкого масштабування.

3.6.	Приклади програмної реалізації
	Архітектурні принципи демонструються через Spotify Web API. Принципи архітектури чітко простежуються в дизайні Spotify Web API. Він побудований на принципах REST та використовує стандарт OAuth 2.0 для безпеки. Кожен запит до API проходить через шар оркестрації, який збирає дані з різних мікросервісів і повертає цілісну відповідь клієнту. На рівні ШІ система використовує обробку даних у TensorFlow для створення векторних представлень вподобань користувача, що демонструє інтеграцію складних обчислювальних моделей безпосередньо в сервісну архітектуру.
